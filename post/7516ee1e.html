Cookie/Session

Cookie

是服务器通知客户端保存键值对的一种技术，每个 Cookie 的大小不能超过 4k

Cookie 不支持中文和一些特殊符号，需要经过 Base64 编码

Cookie 的创建

  • 客户端创建 Cookie 对象，然后通知客户端保存 Cookie
  • 通过响应头 Set-Cookie 通知客户端保存 Cookie
  • 客户端接到响应头后发现有 Set-Cookie，就会检查有没有这个 Cookie，没有就创建，有就修改

服务器如何获取 Cookie

通过 request.getCookies() 方法，返回一个 Cookie[] 数组

只能获取全部的 Cookie，不能指定，如果想要知道特定的 Cookie，只能通过循环判断


Cookie 值的修改

（1）方法一

  • 先创建一个要修改的同名的 Cookie 对象
  • 在构造器同时赋予新的 Cookie
  • 调用 response.addCookie( Cookie )

（2）方法二

  • 先查找到需要的 Cookie 对象
  • 调用 setValue() 方法赋予新的 Cookie 值
  • 调用 response.addCookie( Cookie )

Cookie的生命控制

管理 Cookie 什么时候被销毁，即存活范围，通过 setMaxAge() 方法，默认值为 -1

  • 正数 – 表示在指定的秒数后删除
  • 负数 – 表示浏览器在关闭时立即被删除
  • 0 – 表示立即删除

Cookie有效路径 Path

Cookie 的 Path 属性可以有效的过滤需要发送给服务器的 Cookie

通过 getPath() 方法设置，用请求的地址来进行有效的过滤，默认是到工程路径

  • CookieA path=/工程路径

  • CookieB path=/工程路径/abc

请求地址如下

  1. http://localhost:port/工程路径/a.html
    • CookieA 发送
    • CookieB 不发送
  2. http://localhost:port/工程路径/abc/a.html
    • CookieA 和 CookieB 都发送

利用 Cookie 实现无输入登录

  1. 先创建 LoginServlet，实现登录的业务操作，并且把用户名存储到 Cookie 中
  2. 在登陆页面 login.jsp 中为 username 设置为 Cookie 的值

Session

1、Session 就是一个接口（HttpSession）

2、Session 就是会话。它是用来维护一个客户端和服务器之间关联的一种技术

3、每个客户端都有自己的一个 Session 会话


经常使用 Session 来保存用户登录之后的信息

  • Cookie 保存在客户端（浏览器端）
  • Session 则是保存在服务器端

创建 / 获取

创建和获取 Session 的API是一样的，都是 request.getSession()

第一次调用是：创建 Session 会话

之后调用都是：获取前面创建好的 Session 会话对象

判断是否是初次创建：isNew()

true – 表示刚创建

false – 表示在获取之前已经创建


每个会话都有一个唯一的 id，通过 getId() 方法获取 Session 会话的 id 值


Session 生命周期控制

  • setMaxInactiveInterval()：设置 Session 的超时时间，超过指定的时长，Session 就会被销毁
  • getMaxInactiveInterval()：获取 Session 的超时时长

Session 的默认时长为 30 分钟，可以在 Tomcat 服务器的配置文件 web.xml 中修改配置信息


session 的超时时间指的是，客户端两次请求的最大间隔时长

负数表示永不超时

invalidate() – 表示设置为立即超时无效

Cookie 和 Session

服务器每次创建一个 Session 会话的时候，都会创建一个 Cookie 对象

这个对象的 key 为 JSESSIONID，value 为 新创建出来的 Session 的 id


之后有了 Cookie 之后，每次请求客户端都会把 Session 的 id 以 Cookie 的形式发送给服务器

有了 Session 的唯一标识 id，就可以在内存中找到相应的 Session 并获取信息


当删除掉 Session 的 Cookie 后，服务器因为无法得到 id 值，所以会再次创建一个新的 Session 会话对象，就算没有超过超时时间也会创建一个新的 Session


Filter 过滤器

是 JavaWeb 的三大组件之一，作用是拦截请求、过滤响应

使用步骤

  • 先创建一个类实现 Filter 接口，并完成相应方法
    • 其中 doFilter() 方法是用来完成拦截，进行权限检查
  • 再在 web.xml 中进行配置

生命周期

包含几个方法

  1. 构造器方法

  2. init 初始化方法

  3. doFilter 过滤方法

  4. destory 销毁

第一和第二步在 web 工程启动的时候执行

第三步在每次拦截到请求时都会执行

第四步在 web 工程关闭时执行，同时也会被销毁


FilterConfig

是 Filter 过滤器的配置文件类，每次在创建 Filter 时都会同时创建一个 FilterConfih 类

包含了配置文件的配置信息，主要作用：

  • 获取 Filter 的名称
  • 获取在 Filter 中配置的 init-param 初始化参数
  • 获取 ServletContext 对象

FilterChain 过滤器链

FilterChain.doFilter() 方法的作用

  1. 执行下一个 Filter 过滤器（如果有的话）
  2. 执行目标资源（没有更进一步的拦截检查）

上图的执行顺序实际上为：

前置代码1 –> 前置代码2 –> 执行目标资源 –> 后置代码2 –> 后置代码1


多个过滤器的执行顺序：

  • 注解配置：按照类名的字符串比较规则比较，值小的先执行
    • 如 AFilter 和 BFilter, AFilter 就先执行了
  • web.xml 配置：谁定义在上边，谁先执行

多个过滤器的执行特点：

  1. 所有 filter 和目标资源默认都执行在同一个线程中
  2. 多个 filter 共同执行时（包括目标资源），共用一个 request 对象

Filter 拦截路径

（1）精确匹配

<url-pattern>/target.jsp</url-pattern>

表示请求地址必须为 http://ip:port/工程路径/target.jsp


（2）目录匹配

<url-pattern>/admin/*</url-pattern>

表示请求地址为 http://ip:port/工程路径/admin 目录下的所有内容


（3）后缀名匹配

<url-pattern>*.jsp</url-pattern>

表示请求地址必须以 .jsp 结尾，不能以斜杠开始，且后缀可以是任何字符串，比如 .abc 也可以

Filter 只关心地址是否正确，不关心资源是否存在


ThreadLocal

该类提供了线程局部变量，可以解决多线程的安全问题

具体地说，ThreadLocal 可以给当前线程关联一个数据（可以是变量、对象、数组、集合等）

特点：

  1. 可以为当前线程绑定一个数据（就像 Map 一样可以存取数据，但是 key 为当前线程
  2. 每一个 ThreadLocal 都只能为当前线程关联一个数据
  3. 每个 ThreadLocal 对象创建的时候，一般都是 static 类型
  4. ThreadLocal 中保存的数据，在线程结束后，由 JVM 自动释放
